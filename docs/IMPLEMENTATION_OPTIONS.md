# Implementation Options — Gpt2Whatever

> 本项目已从 "LLM 输出转换器" pivot 为 "可移植 Kimi-Codex 工作流套件"。以下方案围绕新方向展开。

---

## 方案 A：Copyable Scripts + Skills

**描述**
将 Kimi-Codex 工作流所需的所有文件（PowerShell 脚本、skill 模板、目录结构说明）打包成一组可直接复制粘贴到目标项目的文件。无需 Python 运行时，只需 `cp -r` 或解压即可。

| 维度 | 评估 |
|---|---|
| **优点** | 零依赖；任何平台都可用；非开发者也能操作；极易审计 |
| **缺点** | 无参数化能力；不同项目需手动改配置；无版本管理；重复劳动 |
| **适合情况** | 一次性初始化、对 Python 有排斥的环境、极简单的项目 |
| **实现复杂度** | ⭐ 低 |
| **推荐程度** | ★★★☆☆ **可用，但建议配合方案 B** |

---

## 方案 B：Python CLI Installer（推荐）

**描述**
基于当前 Python CLI 骨架，提供参数化配置生成和（未来）一键安装能力。用户通过 `gpt2whatever --project-name X` 生成并预览项目工作流文件，确认后执行安装。配置和模板内嵌在 Python 包中，可版本化。

| 维度 | 评估 |
|---|---|
| **优点** | 参数化（project name, test command, tier defaults）；可测试；可版本发布；可扩展为真实安装器 |
| **缺点** | 需要 Python 3.10+；比纯复制多一层抽象 |
| **适合情况** | 开发者日常在多个项目间复用工作流；需要可维护、可演进的方案 |
| **实现复杂度** | ⭐⭐ 中低 |
| **推荐程度** | ★★★★★ **主推方向** |

---

## 方案 C：Plugin / Marketplace Package

**描述**
将工作流套件扩展为可插拔的插件系统：支持自定义 skill 市场、远程模板仓库、多工作流切换（例如 Kimi-Codex、Claude-Review、Auto-Dev 等）。安装器可下载并合并远程模板。

| 维度 | 评估 |
|---|---|
| **优点** | 极强扩展性；社区可贡献模板；远程更新 |
| **缺点** | 过度工程；需要网络、缓存、版本冲突处理；MVP 阶段完全不需要 |
| **适合情况** | 项目成熟后，面向团队/组织级推广 |
| **实现复杂度** | ⭐⭐⭐⭐ 高 |
| **推荐程度** | ★★☆☆☆ **远期愿景，当前不建议投入** |

---

## 综合建议

1. **Round 3~4（MVP）**：以 **方案 B** 为核心，完成 CLI 的参数化配置生成和预览能力。保留 **方案 A** 的静态文件副本（tools/ 下的脚本）作为兜底。
2. **Round 5+（验证需求后）**：加入 `--install` 标志执行真实文件系统写入，将方案 A 的静态文件内嵌到 Python 包中作为安装源。
3. **远期**：若需要多工作流支持，再评估是否向方案 C 演进。
